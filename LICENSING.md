## Why the eBPF layer is GPL-2.0

The agent loads eBPF programs into the Linux kernel to perform tracing
(TCP events, file I/O, log discovery, etc.). Many of the kernel helper
functions these programs rely on are marked `GPL-only` by the kernel —
the in-kernel eBPF verifier rejects programs that use those helpers
unless the program itself declares a GPL-compatible license via:

```c
char _license[] SEC("license") = "GPL";
```

For this reason the eBPF C sources under `ebpftracer/ebpf/` are
licensed under **GPL-2.0**.

The user-space Go code that loads, manages, and reads from these
programs is licensed under **AGPL-3.0** (as part of the combined
work). Coroot's original contributions to that user-space code
remain under Apache-2.0 — see the table at the top of this file.

## How they are distributed together

The eBPF object code is compiled separately and embedded into the Go
binary as a byte slice (see `ebpftracer/ebpf.go`, generated by
`bpf2go`). At runtime the user-space program loads this byte slice
into the kernel via the `bpf()` syscall — the GPL-2.0 program runs in
kernel space, the AGPL-3.0 user-space program runs in user space, and
they communicate via standard kernel interfaces (maps, perf buffers,
ring buffers).

This is the same arrangement used by every major eBPF tracing agent
(e.g. Cilium, Pixie, Parca, Grafana Beyla, Pyroscope), and follows the
convention established by the kernel community: GPL-licensed eBPF
programs distributed alongside differently-licensed user-space loaders
do not cause the user-space code to become a derivative work of the
GPL kernel.
